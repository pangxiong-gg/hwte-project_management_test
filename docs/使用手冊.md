## 一、系統簡介

數位化專案管理系統是一款專為軟體開發團隊設計的專案管理工具，支援瀑布式（Waterfall）、敏捷式（Agile）、混合型（Hybrid）三種管理模式。系統涵蓋專案管理、需求追蹤、任務分配、Bug 管理、測試管理、資源監控、報表分析等完整功能。

**技術棧**：Vue 3 + TypeScript + Naive UI（前端）/ Express + Prisma + SQLite（後端）

**預設帳號**（由 ADMIN 建立）：

## 二、角色與權限

系統設計四種角色，權限逐層遞減：

### ADMIN（系統管理員）

- **系統管理**：用戶創建/編輯/刪除、角色分配、帳號狀態管理
- **全專案可見**：所有專案的數據與報表
- **全功能訪問**：無任何功能限制
- **數據權限**：資源負載可查看全團隊，報表覆蓋所有專案

### PROJECT_MANAGER（專案經理）

- **專案管理**：創建/編輯專案、選擇管理模式（瀑布/敏捷/混合）、配置階段
- **需求管理**：創建/編輯/刪除需求、追蹤變更
- **任務管理**：創建/指派/編輯任務、設定優先級與截止日期
- **Sprint 管理**：規劃 Sprint、分配任務、追蹤燃盡圖
- **Bug 管理**：創建/指派 Bug、設定嚴重度
- **測試管理**：創建測試用例、指派測試任務
- **資源監控**：查看全團隊負載、識別超載成員
- **文件管理**：上傳/下載/刪除專案文件
- **報表查看**：專案進度、團隊效能、Bug 趨勢、覆蓋率統計

### DEVELOPER（開發人員）

- **任務執行**：查看被指派的任務、更新狀態（待辦→進行中→已完成）、填報工時
- **需求查看**：閱讀專案需求規格
- **程式碼追溯**：關聯 Git branch / commit / PR 到任務
- **Bug 修復**：查看被指派的 Bug、更新修復狀態
- **子任務**：分解大任務為子任務
- **評論討論**：在任務/需求/Bug 下發起討論、@提及隊友
- **Sprint 參與**：查看所屬 Sprint 的任務與進度
- **資源查看**：僅查看自己的負載數據

### TESTER（測試人員）

- **測試執行**：查看測試用例、記錄執行結果（通過/失敗/阻塞）
- **測試用例管理**：創建/編輯測試用例（與 PM 相同權限）
- **Bug 回報**：創建 Bug、設定嚴重度與優先級、指派修復人
- **任務執行**：查看被指派的測試任務、更新狀態
- **需求查看**：閱讀需求以設計測試策略
- **資源查看**：僅查看自己的負載數據

## 三、專案生命週期流程

以下為一個典型軟體專案從起草到結束的完整流程，標註每個階段的主要操作者與功能：

### 階段 1：專案啟動（PM + ADMIN）

**操作者**：PROJECT_MANAGER（主導）+ ADMIN（建立帳號）

1. **建立團隊帳號**
   - ADMIN 登入 → 用戶管理 → 新增用戶
   - 為每位團隊成員建立帳號並分配角色

2. **創建專案**
   - PM 登入 → 專案總覽 → 新增專案
   - 填寫：專案名稱、專案代碼、描述、管理模式、起止日期
   - **管理模式選擇**：
     - **瀑布式**：適合需求明確、階段分明的專案，系統自動生成需求分析→設計→開發→測試→上線五階段
     - **敏捷式**：適合需求迭代、快速交付的專案，以 Sprint 為單位推進
     - **混合型**：瀑布框架 + Sprint 迭代，大型專案推薦

3. **配置 GitHub 整合（選配）**
   - PM 進入專案詳情 → 設定 GitHub Repo 關聯
   - 系統自動接收 push / PR / CI 狀態事件

### 階段 2：需求規劃（PM）

**操作者**：PROJECT_MANAGER

1. **創建需求**
   - 專案詳情 → 需求 Tab → 新增需求
   - 填寫：需求標題、描述、優先級（高/中/低）、狀態（草稿/已確認/已變更）
   - 為需求添加標籤分類

2. **需求變更追蹤**
   - 當需求內容被修改時，系統自動記錄變更歷史
   - 變更記錄包含：修改人、修改時間、修改前後內容對比

3. **上傳需求文件**
   - 專案詳情 → 文件 Tab → 上傳文件
   - 支援規格書、設計稿、會議紀錄等附件

### 階段 3：任務分解與指派（PM）

**操作者**：PROJECT_MANAGER

1. **創建任務**
   - 專案詳情 → 任務 Tab → 新增任務
   - 填寫：任務標題、描述、任務類型（開發/設計/測試/文檔）、優先級
   - 指派負責人、設定計劃工時、截止日期
   - 關聯需求（選填）

2. **子任務分解**
   - 對於大型任務，可創建子任務（parentId 關聯）
   - 當所有子任務完成時，父任務自動標記完成

3. **看板管理**
   - 專案詳情 → 任務 Tab → 看板視圖
   - 拖拽任務卡片改變狀態（待辦 → 進行中 → 已完成）
   - 瀑布模式專案會按階段分組顯示

### 階段 4：Sprint 規劃（敏捷/混合模式）

**操作者**：PROJECT_MANAGER

1. **創建 Sprint**
   - 專案詳情 → Sprint Tab → 新增 Sprint
   - 設定：Sprint 名稱、目標、開始/結束日期

2. **分配任務到 Sprint**
   - Sprint 看板 → 將待辦任務拖入當前 Sprint
   - 系統自動計算 Sprint 總工時與燃盡圖基線

3. **啟動 Sprint**
   - 點擊「開始 Sprint」
   - 團隊成員開始執行 Sprint 內任務
   - 燃盡圖自動追蹤實際剩餘工時 vs 理想線

### 階段 5：開發執行（DEVELOPER）

**操作者**：DEVELOPER

1. **查看任務**
   - 登入後進入「我的任務」頁面
   - 查看跨專案的所有被指派的任務
   - 可切換列表視圖或看板視圖

2. **執行任務**
   - 點擊任務 → 更新狀態為「進行中」
   - 填報實際工時（actualHours）
   - 關聯程式碼：填寫 Git branch、commit、PR 連結

3. **程式碼提交觸發**
   - 開發人員 push 代碼到 GitHub
   - Webhook 自動接收 push 事件
   - 系統自動將 commit 關聯到對應任務

4. **討論與協作**
   - 在任務下發起討論、回覆他人評論
   - 使用 @提及 功能通知隊友
   - 子任務進度會影響父任務狀態

### 階段 6：測試執行（TESTER）

**操作者**：TESTER + PROJECT_MANAGER

1. **設計測試用例**
   - TESTER 進入專案詳情 → 測試 Tab
   - 創建測試用例：標題、前置條件、步驟、預期結果
   - 指派測試執行人

2. **執行測試**
   - 點擊測試用例 → 記錄執行結果
   - 結果：通過（Passed）/ 失敗（Failed）/ 阻塞（Blocked）
   - 填寫實際結果說明

3. **Bug 回報**
   - 測試失敗時創建 Bug
   - 填寫：標題、描述、重現步驟、嚴重度（致命/嚴重/一般/輕微）、優先級
   - 指派給開發人員修復

4. **Bug 修復追蹤**
   - DEVELOPER 收到 Bug 指派通知
   - 修復後更新狀態為「已修復」
   - TESTER 驗證後標記為「已關閉」

### 階段 7：階段推進（瀑布/混合模式）

**操作者**：PROJECT_MANAGER

1. **階段完成檢查**
   - 專案詳情 → 時間線 Tab
   - 查看當前階段的所有任務完成情況
   - 確認里程碑是否達成

2. **推進下一階段**
   - 點擊「推進階段」按鈕
   - 系統自動將專案推進到下一階段
   - 甘特圖更新階段標記

### 階段 8：資源監控與調度（PM）

**操作者**：PROJECT_MANAGER

1. **查看資源負載**
   - 資源負載頁面
   - 三種視圖：
     - **成員卡片**：每位成員的任務數、工時、超載警示
     - **時間軸**：14 天 Gantt 圖，查看任務分佈與並行關係
     - **負載圖表**：柱狀圖比較計劃工時 vs 實際工時

2. **識別瓶頸**
   - 紅色標記：超載（>100% 負載）
   - 黃色標記：警告（80%-100% 負載）
   - 未指派任務列表

3. **調整指派**
   - 將超載成員的任務重新指派給負載較輕的成員

### 階段 9：報表分析（PM + ADMIN）

**操作者**：PROJECT_MANAGER（自己專案）/ ADMIN（全部專案）

1. **專案進度報表**
   - 專案總覽頁面 → 報表區域
   - 柱狀圖：任務完成率、需求完成率
   - 按專案/時間維度篩選

2. **團隊效能報表**
   - 雷達圖：每位成員的多維度效能（任務完成數、準時率、工時填報率、Bug 修復數、測試覆蓋率）

3. **Bug 趨勢報表**
   - 折線圖：30 天內 Bug 發現 vs 修復趨勢
   - 餅圖：Bug 嚴重度分佈

4. **數據導出**
   - 支援 CSV / Excel 格式導出

### 階段 10：專案結案（PM）

**操作者**：PROJECT_MANAGER

1. **確認所有任務完成**
   - 專案詳情 → 任務 Tab
   - 確認所有任務狀態為「已完成」
   - 確認所有 Bug 已關閉

2. **最終報表**
   - 導出完整專案報表存檔

3. **關閉 Sprint（敏捷模式）**
   - 將最後一個 Sprint 標記為完成
   - 回顧燃盡圖數據

4. **更新專案狀態**
   - 將專案狀態更新為「已完成」

## 六、常見問題

**Q1：為什麼我看不到某些功能？**
A：系統根據角色控制功能權限。例如只有 ADMIN 能看到「用戶管理」，只有 PM 能創建專案。請聯繫 ADMIN 確認您的角色設定。

**Q2：如何關聯 GitHub 倉庫？**
A：PM 進入專案詳情 → 設定區域 → 配置 GitHub Repo 名稱（格式：`owner/repo`）。配置後系統會自動接收 push / PR / CI 事件。

**Q3：任務狀態如何流轉？**
A：任務有三種狀態：TODO（待辦）→ IN_PROGRESS（進行中）→ DONE（已完成）。可在看板中拖拽切換，或點擊任務卡片手動更新。

**Q4：子任務如何自動完成？**
A：當一個任務的所有子任務都標記為「已完成」時，父任務會自動標記完成。這適用於多層級的子任務樹。

**Q5：如何收到郵件通知？**
A：進入個人資料頁面 → 開啟「郵件通知」開關。當任務被指派給您、或狀態變更時，系統會發送郵件通知。

**Q6：燃盡圖如何計算？**
A：燃盡圖基於 Sprint 中所有任務的 plannedHours 總和。理想線是從總工時到 0 的線性遞減。實際線根據每日已完成任務的 plannedHours 累計減少。

**Q7：全局搜索支持哪些內容？**
A：支持搜索任務（標題/描述）、需求（標題/描述）、Bug（標題/描述）、專案（名稱/代碼）。在頂部導航欄輸入關鍵字後按 Enter。

**Q8：文件上傳有大小限制嗎？**
A：單個文件上限 50MB。文件以 Base64 格式傳輸並存儲在伺服器本地文件系統。

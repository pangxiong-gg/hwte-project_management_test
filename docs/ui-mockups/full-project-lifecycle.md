# 完整全週期專案流程

## 案例：「客戶門戶網站改版」專案（PROJ-2026-002）

> 以真實專案為例，展示系統如何支撐從**立項 → 需求 → 設計 → 開發 → 測試 → 發布 → 運維**的完整週期。

---

## Phase 0：專案立項

### 第 1 週：專案申請

```
[業務部門 林經理]
    │
    ├──→ 在系統提交「專案立項申請」
    │       └──→ 專案名稱：客戶門戶網站改版
    │       └──→ 業務背景：現有網站使用 5 年，技術老舊，客戶抱怨操作不便
    │       └──→ 核心目標：
    │           ├── 提升客戶滿意度 30%
    │           ├── 支援手機端瀏覽
    │           └── 整合線上支付功能
    │       └──→ 預算：80 萬
    │       └──→ 預計週期：3 個月（2026/06/01 - 2026/08/31）
    │       └──→ 干係人：業務部林經理、IT部陳主管、財務部王會計
    │
    └──→ 提交後狀態：「草稿」
            └──→ 點擊「提交審核」
            └──→ 狀態變為：「審批中」
            └──→ PMO 收到通知：「新專案立項申請待審批」
```

### 第 1 週：立項審批

```
[PMO 王小明]
    │
    ├──→ 收到審批通知
    │       └──→ 點擊查看專案申請詳情
    │       └──→ 審核預算、週期、目標合理性
    │
    ├──→ 指派專案經理：李小華
    │
    └──→ 點擊「核准立項」
            └──→ 狀態變為：「已核准」
            └──→ 自動建立專案：PROJ-2026-002
            └──→ 林經理、李小華收到通知
            └──→ 專案進入「規劃中」狀態
```

### 第 1 週：組建團隊

```
[PM 李小華]
    │
    ├──→ 進入專案詳情 →「團隊」Tab
    │
    ├──→ 添加成員：
    │       ├── 產品負責人：張小芳
    │       ├── 架構師：陳大明
    │       ├── 前端開發：王小花
    │       ├── 後端開發：陳小明
    │       ├── 測試工程師：李小強
    │       └── 運維工程師：張大偉
    │
    └──→ 設定角色權限：
            ├── PM：全部權限
            ├── 開發：任務操作、程式碼提交
            ├── 測試：測試用例、Bug 操作
            └── 運維：發布管理、事件處理
```

## Phase 1：需求管理（第 2-3 週）

### 需求收集與分析

```
[產品負責人 張小芳]
    │
    ├──→ 進入 PROJ-2026-002 →「需求」Tab
    │
    ├──→ 建立業務需求（高層）：
    │       └──→ REQ-001：「客戶可透過手機瀏覽網站」
    │       └──→ REQ-002：「客戶可線上繳費」
    │       └──→ REQ-003：「客戶可查詢歷史訂單」
    │
    ├──→ 拆解使用者需求：
    │       └──→ REQ-001-1：「網站需響應式設計」
    │       └──→ REQ-001-2：「觸控操作優化」
    │       └──→ REQ-002-1：「整合第三方支付」
    │       └──→ REQ-002-2：「支付結果通知」
    │
    └──→ 建立功能需求：
            └──→ REQ-001-1-1：「首頁佈局自適應螢幕寬度」
            └──→ REQ-001-1-2：「導航列折疊為漢堡選單」
            └──→ REQ-002-1-1：「串接綠界金流 API」
            └──→ REQ-002-1-2：「交易紀錄寫入資料庫」
```

### 需求評審與基線化

```
[產品負責人 張小芳]
    │
    ├──→ 選擇一批需求 → 點擊「發起評審」
    │       └──→ 選擇評審人：架構師陳大明 + PM 李小華
    │       └──→ 設定評審截止日：3 天後
    │
    ├──→ 評審人收到通知
    │       └──→ 陳大明：「架構可行，但綠界金流需額外安全評估」
    │       └──→ 李小華：「同意，建議將 REQ-002 分為兩期實現」
    │
    ├──→ 張小芳根據評審意見修改需求
    │       └──→ 調整 REQ-002 範圍
    │       └──→ 補充安全需求 REQ-004
    │
    └──→ 點擊「基線化」
            └──→ 鎖定需求版本
            └──→ 所有需求狀態變為「已基線」
            └──→ 基線後的變更需走「變更申請」流程
```

## Phase 3：開發管理（第 5-10 週）

### Sprint 規劃

```
[PM 李小華]
    │
    ├──→ 建立 Sprint 1（第 5-6 週）
    │       └──→ 目標：完成手機版首頁與導航
    │       └──→ 選入需求：REQ-001-1-1, REQ-001-1-2
    │       └──→ 團隊容量：5 人 × 2 週 = 200 小時
    │
    └──→ 建立 Sprint 2（第 7-8 週）
            └──→ 目標：完成支付功能核心
            └──→ 選入需求：REQ-002-1-1, REQ-002-1-2
```

### 任務拆解與分配

```
[PM 李小華]
    │
    ├──→ 進入 Sprint 1 →「任務看板」
    │
    ├──→ 建立任務：
    │       └──→ TASK-001：「實作響應式 CSS Grid 佈局」
    │           ├── 指派給：王小花
    │           ├── 預估：16 小時
    │           └── 關聯需求：REQ-001-1-1
    │       └──→ TASK-002：「實作漢堡選單元件」
    │           ├── 指派給：王小花
    │           ├── 預估：12 小時
    │           └── 關聯需求：REQ-001-1-2
    │       └──→ TASK-003：「建立響應式圖片載入機制」
    │           ├── 指派給：陳小明
    │           ├── 預估：8 小時
    │           └── 關聯需求：REQ-001-1-1
    │
    └──→ 王小花、陳小明收到通知
```

### 開發進行中

```
[前端開發 王小花]
    │
    ├──→ 點擊 TASK-001 →「開始執行」
    │       └──→ 狀態變為：進行中
    │       └──→ 系統記錄開始時間
    │
    ├──→ 開發過程中：
    │       └──→ Git commit：「feat: #TASK-001 實作響應式 Grid 佈局」
    │       └──→ 系統自動關聯程式碼提交到任務
    │       └──→ 在任務頁面可看到所有 commit 記錄
    │
    ├──→ 每日更新進度：
    │       └──→ 輸入實際工時：6 小時
    │       └──→ 添加備註：「已完成桌面版，手機版進行中」
    │
    └──→ 完成後：
            └──→ 點擊「提交 Code Review」
            └──→ 狀態變為：Code Review
            └──→ 指派審查人：陳大明
            └──→ 陳大明收到通知
```

### Code Review

```
[架構師 陳大明]
    │
    ├──→ 收到通知：「TASK-001 待你 Code Review」
    │
    ├──→ 點擊查看
    │       └──→ 看到關聯的程式碼提交記錄
    │       └──→ 逐行審查，添加評論：
    │           └──→ 「第 45 行：建議加上錯誤處理」
    │           └──→ 「整體結構清晰，變數命名良好」
    │
    ├──→ 點擊「要求修改」
    │       └──→ 王小花收到通知：「TASK-001 CR 未通過，請查看評論」
    │
    └──→ 王小花修改後重新提交
            └──→ 陳大明確認 → 點擊「通過」
            └──→ 狀態變為：測試中
            └──→ 李小強收到通知：「TASK-001 已通過 CR，請測試」
```

## Phase 4：測試管理（第 9-11 週）

### 測試計劃

```
[測試工程師 李小強]
    │
    ├──→ 進入 PROJ-2026-002 →「測試」Tab
    │
    ├──→ 建立測試計劃：「Sprint 1 功能測試」
    │       └──→ 測試範圍：TASK-001, TASK-002, TASK-003
    │       └──→ 測試環境：Staging
    │       └──→ 預計執行時間：3 天
    │
    └──→ 建立測試用例：
            └──→ TC-001：「桌面版 1920px 寬度下首頁正確顯示」
            └──→ TC-002：「手機版 375px 寬度下導航折疊為漢堡選單」
            └──→ TC-003：「平板版 768px 寬度下佈局正確調整」
            └──→ TC-004：「圖片在慢速網路下正確載入縮圖」
            └──→ 每個測試用例關聯對應需求
```

### 測試執行與 Bug 管理

```
[測試工程師 李小強]
    │
    ├──→ 執行 TC-001：PASS ✅
    │       └──→ 標記實際結果：「符合預期」
    │
    ├──→ 執行 TC-002：FAIL ❌
    │       └──→ 實際結果：「漢堡選單在 iPhone Safari 上無法展開」
    │       └──→ 截圖上傳
    │       └──→ 點擊「產生 Bug」
    │       └──→ 系統自動建立 BUG-001
    │           ├── 標題：「漢堡選單在 Safari 無法展開」
    │           ├── 自動關聯：REQ-001-1-2 + TASK-002 + TC-002
    │           ├── 嚴重程度：HIGH
    │           └── 自動指派給：王小花（TASK-002 開發人員）
    │
    └──→ 王小花收到通知
            └──→ 點擊 BUG-001
            └──→ 看到完整上下文：需求描述 + 設計文件 + 測試步驟 + 截圖
            └──→ 修復後提交 commit：「fix: #BUG-001 Safari 漢堡選單點擊事件」
            └──→ 點擊「標記已修復」
            └──→ 李小強收到通知：「請重新驗證」
            └──→ 李小強重新執行 TC-002：PASS ✅
            └──→ BUG-001 狀態變為：已關閉
```

## 全週期中系統的價值體現

### 追溯鏈的完整性

在任何時間點，點開任一個實體，都能看到完整鏈路：

```
點開 BUG-001「漢堡選單在 Safari 無法展開」
    │
```

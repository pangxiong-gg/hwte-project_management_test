# 使用者操作流程

## 核心使用場景

### 場景 1：PM 建立新專案並分配需求

```
[PM 王小明]
    │
    ├──→ 登入系統
    │       └──→ 輸入帳號密碼 → 進入 Dashboard
    │
    ├──→ 點擊「+ 新增專案」
    │       └──→ 填寫專案代碼、名稱、描述
    │       └──→ 點擊「建立」
    │       └──→ 系統自動跳轉到專案詳情頁
    │
    ├──→ 在專案詳情頁點擊「需求」Tab
    │       └──→ 點擊「+ 新增需求」
    │       └──→ 填寫需求標題、描述、優先級
    │       └──→ 系統自動產生編號 REQ-001
    │       └──→ 點擊「儲存」
    │
    └──→ 需求出現在需求列表中
            └──→ 狀態：草稿 → 點擊「提交審核」
            └──→ 狀態變為：待審核
```

**系統行為：**
- 自動產生唯一編號（REQ-001, REQ-002...）
- 記錄建立人和時間
- 變更狀態時自動寫入變更記錄

---

### 場景 2：需求審核通過後，建立開發任務

```
[技術主管 李小華]
    │
    ├──→ 收到通知：「REQ-001 待你審核」
    │       └──→ 點擊通知 → 跳轉到需求詳情
    │
    ├──→ 審核需求內容
    │       └──→ 點擊「核准」
    │       └──→ 狀態變為：已核准
    │       └──→ PM 收到通知：「REQ-001 已核准」
    │
    └──→ 點擊「任務」Tab → 「+ 新增任務」
            └──→ 填寫任務標題：「設計登入 API」
            └──→ 選擇關聯需求：REQ-001
            └──→ 選擇指派給：開發人員陳小明
            └──→ 設定預估工時：8 小時
            └──→ 點擊「儲存」
            └──→ 系統自動產生編號 TASK-001
            └──→ 陳小明收到通知：「TASK-001 已指派給你」
```

**系統行為：**
- 任務自動關聯需求（建立追溯鏈）
- 被指派人自動收到通知
- 任務出現在該用戶的「待辦」看板中

---

### 場景 3：開發人員執行任務並更新進度

```
[開發人員 陳小明]
    │
    ├──→ 登入系統 → Dashboard
    │       └──→ 看到「我的任務」區塊
    │       └──→ TASK-001「設計登入 API」顯示在「待辦」
    │
    ├──→ 點擊 TASK-001 → 查看詳情
    │       └──→ 看到關聯需求 REQ-001 的完整描述
    │       └──→ 點擊「開始執行」
    │       └──→ 任務狀態變為：進行中
    │       └──→ 開始時間自動記錄
    │
    ├──→ 開發過程中
    │       └──→ 在 Git commit 訊息中寫入「#TASK-001」
    │       └──→ 系統自動將程式碼提交關聯到任務
    │
    └──→ 開發完成
            └──→ 點擊「標記完成」
            └──→ 填寫實際工時：6 小時
            └──→ 任務狀態變為：已完成
            └──→ 技術主管收到通知：「TASK-001 已完成」
```

**系統行為：**
- 任務狀態變更自動記錄時間戳
- 實際工時與預估工時自動對比
- 完成後自動通知相關人員

---

### 場景 4：測試人員執行測試並發現 Bug

```
[測試人員 張小芳]
    │
    ├──→ 看到 TASK-001 已完成
    │       └──→ 點擊「測試」Tab
    │
    ├──→ 執行測試用例 TC-001「正確帳密可登入」
    │       └──→ 結果：PASS
    │       └──→ 標記為通過
    │
    ├──→ 執行測試用例 TC-002「錯誤密碼顯示提示」
    │       └──→ 結果：FAIL
    │       └──→ 填寫實際結果：「沒有顯示錯誤提示」
    │       └──→ 點擊「產生 Bug」
    │
    └──→ 系統自動建立 Bug
            └──→ 編號：BUG-001
            └──→ 自動關聯：REQ-001 + TASK-001 + TC-002
            └──→ 指派給：陳小明（TASK-001 的開發人員）
            └──→ 陳小明收到通知：「BUG-001 已指派給你」
```

**系統行為：**
- Bug 自動關聯原始需求、任務、測試用例
- 預設指派給該任務的開發人員
- 開發人員點開 Bug 即可看到完整上下文

---

### 場景 5：開發人員修復 Bug

```
[開發人員 陳小明]
    │
    ├──→ 收到通知：「BUG-001 已指派給你」
    │       └──→ 點擊通知 → 跳轉到 Bug 詳情
    │
    ├──→ 查看 Bug 詳情
    │       └──→ 看到：
    │           ├── 原始需求：REQ-001「使用者登入功能」
    │           ├── 關聯任務：TASK-001「設計登入 API」
    │           ├── 測試用例：TC-002「錯誤密碼顯示提示」
    │           └── 重現步驟：「輸入錯誤密碼 → 無提示」
    │       └──→ 不需要再問測試人員「這個 Bug 是什麼意思」
    │
    ├──→ 修復程式碼
    │       └──→ Git commit：「fix: #BUG-001 密碼錯誤時顯示提示」
    │
    └──→ 點擊「標記已修復」
            └──→ 狀態變為：已修復
            └──→ 張小芳收到通知：「BUG-001 已修復，請驗證」
```

**系統行為：**
- Bug 詳情頁顯示完整追溯鏈（需求→任務→測試）
- 開發人員一次看懂所有上下文
- 修復後自動通知測試人員驗證

---

### 場景 6：主管查看整體進度

```
[主管 王大明]
    │
    ├──→ 登入系統 → Dashboard
    │       └──→ 看到統計卡片：
    │           ├── 進行中專案：5
    │           ├── 待處理需求：23
    │           ├── 進行中任務：47
    │           └── 未解決 Bug：8
    │
    ├──→ 點擊「專案總覽」
    │       └──→ 看到所有專案列表
    │       └──→ 每個專案顯示需求數/任務數/Bug 數
    │
    ├──→ 點擊「數位化專案管理系統」
    │       └──→ 進入專案詳情
    │       └──→ 看到：
    │           ├── 需求完成度：67%（8/12）
    │           ├── 任務完成度：41%（14/34）
    │           └── Bug 解決率：62%（5/13）
    │
    └──→ 點擊「任務看板」Tab
            └──→ 看到 Kanban 看板
            └──→ 一眼看出哪些任務卡住、誰負責、是否逾期
```

**系統行為：**
- Dashboard 提供全局視圖
- 專案詳情提供鑽取分析
- 看板提供即時狀態
- 所有數據即時更新

---

### 場景 7：需求變更，影響自動追溯

```
[PM 王小明]
    │
    ├──→ 客戶說：「登入功能要加手機驗證」
    │
    ├──→ 點擊 REQ-001「使用者登入功能」
    │       └──→ 點擊「編輯」
    │       └──→ 在描述中加上「需支援手機驗證」
    │       └──→ 點擊「儲存變更」
    │
    └──→ 系統自動顯示影響分析：
            ├── ⚠️ 影響任務：2 個
            │   ├── TASK-001「設計登入 API」→ 需要修改
            │   └── TASK-002「實作登入 UI」→ 需要修改
            ├── ⚠️ 影響測試：1 個
            │   └── TC-002「錯誤密碼顯示提示」→ 需補充測試
            └── ⚠️ 影響版本：v1.0.0 → 可能延後
            
            └──→ 相關人員自動收到通知：
                ├── 李小華：「REQ-001 已變更，影響你負責的任務」
                └── 張小芳：「REQ-001 已變更，需更新測試用例」
```

**系統行為：**
- 自動計算變更影響範圍
- 即時通知所有相關人員
- 歷史版本保留，可追溯每次變更

---

## 通知觸發總覽

| 觸發事件 | 接收人 | 通知內容 |
|----------|--------|----------|
| 需求指派 | 被指派人 | 「REQ-XXX 已指派給你」 |
| 需求變更 | 關聯任務負責人 | 「REQ-XXX 內容已變更，影響你的 TASK-XXX」 |
| 任務指派 | 被指派人 | 「TASK-XXX 已指派給你」 |
| 任務逾期（2天前） | 負責人 + PM | 「TASK-XXX 將在 2 天後到期」 |
| 任務完成 | 任務建立者 + PM | 「TASK-XXX 已完成」 |
| Bug 指派 | 被指派人 | 「BUG-XXX 已指派給你」 |
| Bug 修復 | 測試人員 | 「BUG-XXX 已修復，請驗證」 |
| 測試失敗 | 相關開發人員 | 「TC-XXX 執行失敗，已產生 BUG-XXX」 |
| 審批待處理 | 審批人 | 「REQ-XXX 變更申請待你審批」 |

## 關鍵設計洞察

### 為什麼這些流程能解決 ABC 痛點？

| 痛點 | 對應流程 | 效果 |
|------|----------|------|
| **A. 需求變更影響不明** | 場景 7 | 變更時自動顯示影響的任務、測試、版本 |
| **B. 進度不透明** | 場景 6 | Dashboard 統計 + 專案詳情百分比 + 看板視覺化 |
| **C. Bug 溝通成本高** | 場景 4 + 5 | Bug 自動關聯需求/任務/測試，點開就懂上下文 |

### 減少的溝通成本

**以前（Excel + 郵件）：**
```
PM：「登入要加手機驗證」（郵件）
開發：「收到」（心裡：影響多大？）
測試：「收到」（心裡：要測什麼？）
→ 3 天後：發現 API 改了但文件沒更新
→ 郵件來回 10 封確認影響範圍
```

**現在（系統）：**
```
PM：在系統修改 REQ-001
→ 系統自動顯示：影響 2 個任務、1 個測試用例
→ 開發/測試收到通知，在自己的頁面看到「有東西變了」
→ 沒有人被遺漏
```

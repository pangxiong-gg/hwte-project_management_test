## 問題 3：最痛的環節

**問題：** 你們目前軟體開發過程中，最痛的是哪一個環節？

**選項：**
- A. 需求經常變更，不知道變更影響了哪些程式碼和測試
- B. 開發進度不透明，主管問「功能做好了沒」永遠不知道真實狀態
- C. Bug 太多，開發和測試之間的溝通成本高
- D. 沒有統一的發布流程
- E. 程式碼審查和品質管控缺失
- F. 其他

**使用者選擇：A + B + C**

**收斂結論：**
這三個痛點其實是相互關聯的：
- **A（需求變更影響不明）** → 導致程式碼和測試不同步
- **B（進度不透明）** → 因為需求和任務沒有清晰的對應關係
- **C（Bug溝通成本高）** → 因為 Bug 與需求/程式碼之間缺乏關聯

**核心需求：** 建立「需求-任務-程式碼-測試-Bug」之間的完整追溯鏈，讓變更影響一目瞭然，進度可視化，開發測試在同一個系統中協作。

## 問題 4：團隊規模與現有工具

**問題：** 你們目前的團隊規模和使用的工具有哪些？

**選項：**
- A. 小團隊（5-15人），主要用 Excel / 郵件 / 口頭溝通
- B. 中型團隊（15-50人），用了一些工具但沒整合
- C. 大型團隊（50-200人），有現有系統但不滿意
- D. 其他

**使用者選擇：A**

**收斂結論：**
- 5-15人小團隊，處於「Excel 撐不住」的轉折點
- 沒有統一系統，資訊散落在 Excel、郵件、口頭中
- 需要一個「夠用但不複雜」的系統，學習成本要低
- 系統要能隨團隊成長而擴展（不會半年後又需要換系統）

## 問題 5：技術棧與部署約束

**問題：** 你們對技術棧和部署方式有什麼偏好或限制嗎？

- A. 沒有特別偏好，聽建議

**使用者選擇：E（B + C + D）**

- 綜合來看：需要一個可私有化部署的開源軟體開發專案管理系統

## 設計討論中的誤解與澄清

**AI 的誤解：** 以為使用者在問「管理系統本身的功能如何迭代擴充」

**使用者的真意：** 「迭代」指的是**客戶軟體專案本身的需求變更迭代**——即這個管理系統如何支援「客戶需求在開發過程中不斷變更」的場景

**收斂結論：**
- 回到最初痛點 A「需求變更影響不明」
- 需要設計的是「需求變更管理」機制，而非「系統功能迭代」機制
- 核心問題：需求變了，如何自動/手動標記影響範圍？

## 功能討論：通知系統

**使用者需求：**
- 需求變更時，對應負責人可收到通知（Email / 手機）
- 登入系統時有「通知任務欄」可查看所有待處理和歷史通知

**收斂結論：**
- 通知系統是基礎設施，不是某個 Phase 的功能

- 通知中心需要有「已讀/未讀」區分、歷史記錄、快速跳轉到相關頁面

## 技術討論：為何選用 TypeScript？

**使用者問題：** 為何狀態管理、API 服務層選用 TypeScript？

**回答重點：**
- TypeScript 不是「另一種語言」，是「加了型別檢查的 JavaScript」
- 在狀態管理和 API 層的價值最大：介面定義、自動補全、編譯期錯誤檢查
- 對小團隊尤其重要：減少口頭確認、降低新人上手成本

---

## 技術討論：Vue 可以取代 TypeScript 嗎？

**使用者問題：** Vue 可以取代 TypeScript 嗎？考量後續維護便利性

**核心澄清：**
- Vue 是「框架」，TypeScript 是「帶型別的 JavaScript」
- 兩者不是取代關係，是搭配關係
- Vue 3 官方原生支援 TypeScript
- 不用 TypeScript = 用純 JavaScript 寫 Vue

**關鍵議題：** 長期維護時，「純 JavaScript」vs 「TypeScript」哪個更便利？
